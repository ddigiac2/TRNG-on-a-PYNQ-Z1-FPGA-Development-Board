# ACR30
This section of the repository goes through the process of implementing the TRNG design based on the cellular automata topology rule of CA30 in the Xilinx Vivado application. Following the software setup described in the previous sections and opening the Vivado application, a new project can be created. When creating a new project, choose a name such as “TRNG_design” along with a location on the machine where the project can be saved. Make sure to select the RTL project option and skip through adding any source or constraint files for now. For selecting an AMD part or board, search for “xc7z020clg400-1” and select this part that represents the SoC and hardware used in this tutorial. After clicking ‘Next’, the default part number should match what was entered previously, along with the family being the ‘Zynq-7000” family. Click on ‘Finish’ and now the project space has been created, allowing for addition of source and constraint files. 

The first source file that can be added and will stand as a low-level module, is the ACR30 module. In the project workspace, go to File -> Add Sources -> Add or create Design Sources -> Create File -> File Name: “ACR30” and File Type: “Verilog”. Now this file can be edited with the ACR30 asynchronous circuit implementation, as depicted in Figure 1 below. This module contains the four input wires (previous, pass_capture, set, next) and one output wire (current). The previous and next input wires represent the left and right neighboring cells in cellular automata, respectively. The previous wire is XORed with the result of an OR of the current wire and next wires (next_or_current = current | next  pass = previous ⊕ next_or_current). The output of the XOR gate represents the pass input signal of a 2x1 mux, where the other mux input signal is the current wire. The mux select signal is the pass_capture input wire (mux_pass_capture = (pass_capture) ? pass : current). When set to a ‘1’, this means that pass is enabled and the circuit is in self-oscillation mode. When set to a ‘0’, this means that capture is enabled and the current output is latched. Two inverter gates are applied after the mux output to add delay as a buffer for asynchronous operation and to save power consumption when in the capture state (buf1 = ~mux_pass_capture buf2 = ~buf1). An OR gate is then applied with the output of the buffer and the set wire, whereas when set is a ‘1’ then the circuit is forced to output a ‘1’ (current = set | buf2).

One large issue that can occur when running synthesis in Vivado for this circuit is that the application will try to optimize this asynchronous circuit and therefore will cause it to perform undesirably. To have Vivado not optimize this ACR30 circuit, the line (* KEEP = "TRUE", DONT_TOUCH = "yes" *) must be added before instantiating any of the local wire resources used in this module.

# TRNG_main and All-Zero Detector
With the low-level module of an ACR30 implemented, a full TRNG module can be then created according to the TRNG design in Figure 2. Similar to that of creating the source file for the ACR30, a Verilog source file must be created for TRNG_main. As seen in the figure, there are nine identical ACR30 elements connected in series to its neighboring elements that must be instantiated. The middle ACR30 element does not have its current output sampled by a DFF for the final TRNG output, as it is used solely for a previous and next input to its neighboring elements. The other eight ACR30 elements utilize the same set and pass/capture signal inputs, whereas each of the eight current wire outputs are sampled by a DFF. The enable for this DFF is the pass/capture signal, meaning that when the positive edge of this signal is hit, the TRNG will output the random number byte.

Another low-level module that is significant to the TRNG design is the all-zero detector module depicted in Figure 3. This is needed if the current signals become all zero from the ACR30 elements, as it will stay all zero without oscillation and cause entropy to be zero. When an all-zero detector outputs a ‘1’, this will drive the set signal of the middle ACR30 element and re-trigger oscillation. This module is implemented by pairing two current output wires from ACR30 elements and putting them through a NOR gate (nor1 = ~(ACR30_curr1 | ACR30_curr2) nor2 = ~(ACR30_curr3 | ACR30_curr4)), and then an AND gate (osc_retrigger = nor1 & nor2).

# TRNG Top-Level File and UART Controller
With having a higher-level TRNG module completed, a top-level module can be implemented with the new Verilog source file being TRNG_top. Make sure to right-click on this source file in the project manager workspace and set the file as top. This top-level module instantiates four TRNG_main modules and XORs their 8-bit random number outputs to increase entropy/randomness into a final output byte, also acknowledged as whitening as shown in Figure 4. Similar to that of the ACR30 module with not wanting any optimizations to the logic, the same condition must be added for the internal resources instantiated in the top-level for the XOR wires. Vivado when synthesizing may see the same TRNG_main module being XORed with each other and try to remove the gates when optimizing.

Besides for instantiating and XORing the TRNGs to attain a final random byte of data, the top-level module must include logic for controlling the pass_capture signal that acts as the sampling rate. As an input to the top-level will be the 125 MHz sys_clk the FPGA has, set in the constraints, there is some logic necessary to act as a clock divider for sampling. Triggering the pass_capture signal for all four TRNGs for sampling their outputs at a sufficient frequency that does not affect the baud rate of the UART transmission that is to be implemented, requires some calculations. The UART baud rate will be 115200 and the sys_clk frequency is 125 MHz, meaning that 1085 cycles per duration of sending one bit over UART (125Mhz / 115200 = 1085). To send 10 bits with some extra leeway, the TRNGs can be sampled every 11000 cycles to have a new random byte available. As seen in the appendices of the Verilog code, a counter variable can be incremented every sys_clk positive edge until reaching a value of 11000, in which the pass_capture signal for each of the TRNGs will be driven for capture before being reset for oscillation and passing again. In this same always block and conditional statement, a wire and register can be instantiated (data_available) to make the UART aware that there is a byte of data to be transmitted.

In regard to the UART transmission, another lower-level module can be made from the creation of a Verilog source file. The UART sends a singular bit of data every 1085 cycles, where the eight bits of data to be sent when data_available is high is the output of the four XORed TRNG outputs. The UART transmission consists of a four-state finite state machine (FSM) that starts in an idle state. Once data_available is driven high, the FSM begins to transmit the byte in a bit-by-bit fashion that is separated every 1085 cycles of the sys_clk. Once it has transmitted the eighth bit of the random number, the FSM enters the stop state in which it waits another 1085 clock cycles before returning back to the idle state for the next data_available trigger. For more specifics on the Verilog implementation of the UART transmission module, please see the uart_tx.v file.

The UART module can then be instantiated as a uart_controller in the top-level module with each output bit that is transmitted is assigned to the output wire of the entire logic. This wire will be constrained to an I/O pin on a connector on the board when making a constraints file for the logic. Before setting constraints, the functionality of the set input wire for each of the TRNG_main modules is not necessary for the particular application we are fulfilling, therefore the set inputs can all be tied low to a ‘0’.

# Constraints
Finally, before running the synthesis and implementation for the logic design, the last step that is necessary is creating a constraints file. Similar to that of creating a source file, create a constraints file with the name TRNG_top.xdc. For the clock and singular I/O pin that we need for the TRNG, these constraints can be found from PYNQ-Z1 datasheets and reference manuals. As shown in the constraints file (.xdc), we set and create the 125 MHz on-board system clock to the sys_clk input signal from the top-level module. Also, we set the top left-most pin on the PMOD Header JA (pin Y18) to the transmitted TRNG output bit from the top-level module as well. The remaining constraints to include in the .xdc file are for preventing optimizations and overriding any ring oscillator/combinational loop errors that are created from the asynchronous circuit. Besides for reducing errors to warnings about allowing combinational loops in order for bitstream to generate, we can add a similar line in the constraints file as done in previous source files to not touch/optimize the ACR30 logic. Once all of these modules have been completed along with the constraints, it is time to begin validating the design and generating a bitstream for programming.
